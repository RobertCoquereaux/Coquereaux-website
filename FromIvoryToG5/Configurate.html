How to make a remote backup of your FEP files on a new firewire or USB disk hooked to your G4 or G5 running OSX.
So we want to make the backup on the G5, for instance (or on a disk attached to it).
You launch ShareWayIP on the Mac II FX side and you check the IP number of the mac on the TCP/IP control panel.
You mount your FX disks on your G5 desktop by calling this IP number.
 Unfortunately you will not see the FEP files themselves, because they are invisible. So the idea is to use Resedit on the FX to make them visible. You select your old SCSI disk, ask for information (ResEdit menu) about the MacIvoryFepN files, you uncheck the invisibility box, and leave ResEdit, saving your modification.  Then your MacIvoryFepN files are visible : you can copy them easily to your G5, using the ethernet connection. Finally you store them wherever you like.
However, remember that you have to reverse the process in order to use the FEP file. Indeed for a reason that eludes me, the FEP processor will not mount visible FEP files when you launch the genera application. So you have to use ResEdit again on the FX side, and make your original FEP files  invisible again in order  your Ivory processor to use them.
 
Configure the Ivory for the ethernet connection
This is easy.  First of all you allow the Ivory to use the other ethernet interface (remember that you have two of them on your nubus), this is again done with the MacIvory control panel. Then, once Lisp is running, you Select Activity Namespace Editor and attribute a local IP number to your Ivory machine (not the same as the one used by the FX host). If you are lucky (I mean if you do not have to change the site, the namespace, and everything else) this was probably already done and you don’t have to worry. Of course you also need to install the X11 system on the Ivory side, but this is also standard.  You have also to declare your G5 in the Namespace of the Ivory, so you create the namespace object host G5, if it does not exist already and you give its local IP number (by the way, you should also give it on the G5 side, of course, but you should do it manually, otherwise,  if it gets later changed by DHCP, the connection with the Ivory will cease to work and you would have to use the namespace editor again..)  With the same Namespace Editor, you have also to declare that the G5 offers the following service :  X-Window-System (sevice)TCP (medium) X-window-system (protocol) .Finally, yes, still with the Namespace Editor, you have to declare your Airport Base if you have one (in my case this base works as a gateway on my private network);  in any case, if your G5 is not directly connected by ethernet to the Ivory (and to the FX), you have to do something... In my case I declared the Airport Base as if it were a computer, I gave its IP to the namespace and added the fact that it offers a service gateway (service) IP (medieum) internet-gateway (protocol).
Now you can check that everything is OK.
You do a ping from the G5 on the  IP number of the Ivory. If it works... you are happy.
You select the activity Peek, on the Ivory side, select network, and check that your network is alive. Actually, I can see that my Airport Base is “alive”.   I don’t see the G5 with Peek,  at that moment (it will appear later).
If your network is blank, try this (it may not be needed). From the Ivory Lisp listener, or from Peek, you type two commands : Reset Network, and Enable Network.
Now you try to start an X screen on your G5 running Tiger, by typing the command Start X Screen G5 from the Lisp Listener.
And you fail for three reasons. The first two are obvious, the last one is not.
 
Last X11 things to do on the G5 side :
1)  You go to the G5 and start X11 (maybe you had forgotten to launch it ! Of course starting the Terminal application is not sufficient)
2) From X11 you start a xterm and issue the command xhost + (the IP number of the Ivory)
These were the obvious two reasons for which you had failed. Now, about the last :
The file /Library/Preferences/com.apple.x11.plist is automatically updated by the most recent versions of Tiger - I have version 10.4.8 - actually, with the last versions of Apple X11, without telling the users, of course !  This was not the case with the previous versions of OSX.
This file is a binary file, so  it is not obvious to discover that the probem is indeed there. You cannot read and edit this file directly from the command line, but you can do it with  the Property List Editor.  And you will notice that a property called nolisten_tcp of Class Boolean has been set to value Yes. I believe that this is for security reasons (people at Apple believe that only ssh is used, nowadays !). In any case, this prevents the tcp connection to work, even if you had issued a xhost + command.  The problem is easy to fix (once you know): just use the Apple Property List Editor and replace Yes by No. It means that from now on you will not no-listen tcp, so you will listen.   Remember that next Apple update of Tiger or X11 may change this plist again without telling you.
